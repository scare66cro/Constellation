## What was imaged
First **32 GiB** of `/dev/nvme0n1` (`bs=4M count=8192`) — covers the MBR plus
all three partitions. The empty tail is omitted to keep the image small;
because the table is MBR, the snapshot is complete and bootable on its own.

## Method (clean snapshot)
1. `sudo systemctl stop agristar-bridge uisvelte` + `sync` (quiesce writers)
2. `sudo dd if=/dev/nvme0n1 bs=4M count=8192 | gzip -1` → streamed over SSH to the dev box
3. `sudo systemctl start agristar-bridge uisvelte`

Bridge downtime ≈ 10 min. Reproduce with `image_pi5.sh` (reads the sudo
password from a local file so it never enters logs).

## Restore
On a target Pi5 with a ≥32 GB NVMe (booted from other media or via a reader):

```bash
# 1) verify the archive first
gzip -t pi5-nvme-20260605-1610.img.gz

# 2) write it back (DESTROYS the target disk — double-check the device!)
gunzip -c pi5-nvme-20260605-1610.img.gz | sudo dd of=/dev/nvme0n1 bs=4M status=progress conv=fsync

# 3) (optional) reclaim the rest of a larger disk:
#    grow partition 3 (or add one) with `parted`, then `btrfs filesystem resize max <mnt>`
```

No GPT fixup needed (MBR). After writing, the Pi boots with the exact bridge
+ firmware state captured here.
